# nanostores.rs

**Rust-owned reactive state for TypeScript apps**: a thread-safe Rust port of
[nanostores](https://github.com/nanostores/nanostores) plus a wasm-bindgen
bridge that projects Rust stores into *real* JS nanostores stores. State lives
in Rust; the TS side gets ordinary atoms and maps — `@nanostores/react`,
`@nanostores/preact`, `computed`, `listenKeys` and the rest of the ecosystem
work on projections without knowing they are projections. Writes are
bidirectional: `$store.set(v)` in TS goes through Rust (which may transform or
reject the write) and the change flows back into the projection.

## Why

Moving app logic into Rust/wasm usually costs you your reactive UI layer:
state behind a wasm boundary is opaque to JS state managers. nanostores.rs
keeps the source of truth in Rust and mirrors it into nanostores with the
API's own granularity — atoms cross whole, map keys cross as `(key, value)` —
so JS-side key subscriptions keep working natively and no diff engine is
involved.

The core crate is a standalone reactive library: it works in native Rust
programs (no wasm required) and is `Send + Sync` throughout.

## Install

Rust side (your wasm crate; `nanostores` alone for native use):

```toml
[dependencies]
nanostores = "1"
nanostores-wasm = "1"
```

TS side:

```bash
npm install nanostores-wasm nanostores
```

`nanostores` is a peer dependency — the projections must be created by the
same nanostores instance your app uses.

## How it works

One synchronous round trip per write; the projection's real `set` is only
ever called from the wasm change callback, so there is no echo loop and no
double-fire by construction:

```
TS  $count.set(5)
      └─ forwarded to wasm handle.set(JsValue)      (no local set)
           └─ Rust atom.set(5)
                ├─ on_set hooks: may transform the value or abort
                ├─ PartialEq check: equal value -> no notification at all
                └─ Rust listeners fire (computed recompute, your logic, ...)
                     └─ bridge listener serializes -> JS callback
                          └─ projection's ORIGINAL set applies the value
                               └─ JS subscribers fire (React/Preact re-render)
```

- **Rejection**: `on_set` calls `ctx.abort()` → no notification comes back →
  the projection keeps its old value; the TS write becomes a silent no-op
  (deserialize errors additionally land in `console.error`).
- **Transformation**: `on_set` mutates `ctx.value_mut()` → the transformed
  value is what returns to JS.
- **Rust-initiated writes** (`count.set(...)` anywhere in Rust) go through
  the same bridge listener and reach the projection identically.
- **Map granularity**: `setKey` crosses as `(key, value)` and is applied via
  the projection's native `setKey`, so JS `listenKeys` filtering works
  without any bookkeeping.
- **Laziness**: the wasm subscription exists only while the projection has
  JS listeners (nanostores `onMount` / unmount drives it). Unmounted
  projections still write through and re-read `handle.get()` after a write.

## Rust core

```rust
use nanostores::{atom, map, computed, batched, on_set, NanoMap};
use serde::{Deserialize, Serialize};

#[derive(NanoMap, Clone, PartialEq, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
struct User {
    name: String,
    age: u32,
    display_name: Option<String>,
}

let count = atom(0_i64);
let user = map(User { name: "Ada".into(), age: 36, display_name: None });

// computed: eager while mounted, lazy when unmounted
let doubled = computed((count.clone(),), |c| c * 2);

// batched: collapses several changes into one recompute per scheduler tick
let summary = batched((count.clone(), user.clone()), |c, u| {
    format!("{} has {c} clicks", u.name)
});

// subscribe fires immediately + on change; listen only on change.
// Subscription is RAII: drop = unsubscribe, .detach() = keep forever.
let sub = doubled.subscribe(|value, _| println!("doubled = {value}"));

count.set(2);                 // PartialEq skip: equal values don't notify
user.set_age(37);             // typed setter generated by #[derive(NanoMap)]
user.set_key("displayName", serde_json::json!("Ada L."))?; // string-keyed
user.update(|u| u.age += 1);  // notifies all key listeners

// interceptors: transform or reject writes
let _guard = on_set(&count, |ctx| {
    if *ctx.value() < 0 { ctx.abort(); }   // reject
    else { *ctx.value_mut() *= 10; }       // transform
});

drop(sub);
```

Lifecycle hooks mirror JS nanostores: `on_start` / `on_stop` (first listener
attached / last removed), `on_mount` (start + cleanup on stop), `on_set` /
`on_notify` (intercept before write / before listeners, `ctx.abort()`
cancels).

Key listeners for maps (`update()` and whole-value `set` notify with
`changed_key = None`, which matches every key filter — same as JS
`listenKeys`):

```rust
let _sub = user.listen_keys(&["name", "displayName"], |u, changed| {
    println!("{changed:?} -> {}", u.name);
});
```

`computed` / `batched` take a tuple of source stores — any `StoreLike` mix
(atoms, maps, other computeds), arity 1 to 8.

### Threading & scheduler

All stores are cheap-to-clone `Send + Sync` handles (`Arc` inside). Listeners
never run under store locks, so reentrant `set` from inside a listener is
legal. `batched` defers recomputes through a global `Scheduler`:

- native default: immediate (batched behaves like `computed`); install your
  own with `nanostores::set_scheduler`, drain manually with
  `nanostores::flush()` (deterministic tests),
- in the browser the bridge installs a `queueMicrotask` scheduler
  automatically — exact JS `batched` semantics.

## Wasm bridge

Define state in your wasm crate and export it with `export_stores!`:

```rust
use nanostores::{atom, computed, map, Atom, Computed, MapStore, NanoMap};
use std::sync::LazyLock;

static COUNT: LazyLock<Atom<i64>> = LazyLock::new(|| atom(0));
static USER: LazyLock<MapStore<User>> = LazyLock::new(|| map(User::default()));

nanostores_wasm::export_stores! {
    pub fn stores() -> StoreHandles {
        atom count: i64 = LazyLock::force(&COUNT);
        map user: User = LazyLock::force(&USER);
        readable doubled: i64 = LazyLock::force(&DOUBLED);
    }
}

// Rust-initiated mutations reach the UI reactively too:
#[wasm_bindgen]
pub fn increment() {
    let count = LazyLock::force(&COUNT);
    count.set(count.get() + 1);
}
```

The macro emits the typed `.d.ts` contract (`StoreHandles`, `StoreKinds`,
`ProjectedStoreHandles`). After `wasm-pack build`, generate the app-side
wrapper next to the wasm-bindgen output:

```bash
node packages/nanostores-wasm/scripts/generate-wrapper.mjs path/to/pkg/my_core
```

TS side — no manual wasm init, `createStores()` handles it (memoized):

```ts
import { createStores } from "./pkg/my_core_stores";

const stores = await createStores();

stores.count.set(5);                    // -> Rust -> back into the projection
stores.user.setKey("name", "Grace");    // only (key, value) crosses the boundary
listenKeys(stores.user, ["name"], cb);  // native nanostores key subscription
```

Projections are real nanostores stores with lazy wasm subscriptions: no JS
listeners → no boundary traffic (`onMount` / unmount drives the wasm-side
subscription). If Rust rejects a write via `on_set` abort, the projection
keeps its old value and the error lands in `console.error`.

You can also project stores manually without the macro:
`AtomHandle::from_atom`, `AtomHandle::from_readable` (any `StoreLike`, e.g.
`Computed`), `MapHandle::from_map`, and on the TS side `projectAtom` /
`projectReadable` / `projectMap` / `projectStores` from the
`nanostores-wasm` npm package.

### Initialization

Nothing to call in the usual flow: every bridge entry point (`export()`,
handle constructors) installs the microtask scheduler idempotently, and the
generated `createStores()` initializes the wasm module itself. Explicit knobs
for library authors:

- `nanostores_wasm::install_scheduler()` — public, idempotent; call it if
  your Rust wasm code uses `batched` before creating any bridge handles,
- expose your own `#[wasm_bindgen] pub fn init()` calling it, if you want a
  forced init from TS.

**One wasm module per app.** Two separate `.wasm` binaries have two isolated
memories — stores do not cross wasm instances. Ship your Rust library as a
regular Cargo dependency (rlib) compiled into the app's single cdylib;
wasm-bindgen merges its exports into the app's `pkg`.

### Boundary semantics

- values cross via `serde-wasm-bindgen` (plain JS objects, no JSON strings);
  `HashMap` serializes as a plain object,
- `None` ↔ `undefined` (not `null`),
- every crossing creates a fresh JS object — change detection (`PartialEq`
  skip) happens on the Rust side,
- key names follow serde attributes: `#[serde(rename_all = "camelCase")]`,
  `#[serde(rename = "...")]` — Rust and TS agree on strings,
- derive `tsify::Tsify` on your state types to get real TS interfaces at the
  boundary.

### Divergences from JS nanostores (v0.1)

- Rust core has no `STORE_UNMOUNT_DELAY` (no portable timer); the JS
  projection side keeps the native nanostores delay,
- `batched` on native without a scheduler degenerates to `computed`,
- not ported yet: `deepMap` / `setPath`, `effect`, `task` / `allTasks`,
  `keepMount`, `mapCreator`.

## Example app

```bash
# 1. build the wasm core + generate the wrapper
make build-example

# 2. run the UI
npm install
npm run dev --workspace browser-app-ui   # http://localhost:5173
```

Demonstrates: TS write → Rust → projection, Rust write (`increment()`) →
projection, Rust `computed`/`batched` reacting to TS writes, `listenKeys` on
a map projection.

## Development

```bash
make test           # cargo test + clippy (native)
make test-wasm      # wasm32 build + clippy
make test-browser   # wasm-pack tests (headless chrome)
make test-js        # rebuild wasm pkg + typecheck + vitest + production build
make doc            # cargo doc
make release        # interactive: bump version, tag vX.Y.Z, push
```

CI mirrors these targets (native / wasm / browser / js jobs); the browser job
is non-blocking because headless chrome is flaky in CI.

Releases are tag-driven: `make release` bumps one version across the Cargo
workspace and npm packages, commits, tags `vX.Y.Z` and pushes; the tag
triggers `release.yml`, which re-runs CI, creates a GitHub release, and
publishes to crates.io (`nanostores-macros` → `nanostores` →
`nanostores-wasm`) and npm. Requires `CARGO_REGISTRY_TOKEN` and `NPM_TOKEN`
repository secrets.
